Chapter 06 · Compression & local AI

From Giant to Your Desk

A 70-billion parameter model, fully precise, requires 140 GB of memory — more than any consumer machine has. Yet people run these models on laptops. This chapter explains the compression techniques that make that possible, and what you give up when you use them.

~38 min read 11 sections Compression & local AI

The size problem

The chapter on hardware established that model parameters must fit in GPU memory to run. A 7B model at full 32-bit precision requires 28 GB of VRAM. A 13B model requires 52 GB. A 70B model requires 280 GB. A theoretical GPT-4-scale model at 1.8T parameters would require roughly 7,200 GB — 90 H100s just to hold the weights.

Consumer machines top out at 24 GB of GPU VRAM (an NVIDIA RTX 4090), though Apple's unified memory architecture allows Macs with 64–192 GB RAM to run models using that memory pool more efficiently. The gap between "models that exist" and "models that fit on consumer hardware" is enormous.

Two complementary techniques close most of this gap: quantisation (reducing the precision of each parameter) and distillation (training smaller models to approximate larger ones). Together they've made it possible to run genuinely capable models on hardware that costs a few hundred dollars. This democratisation of local inference is one of the most significant developments in AI of the last two years.

Quantisation: rounding down to fit

Every parameter in a neural network is a number. During training, that number is stored at high precision — typically 32-bit floating point (FP32), which can represent values with about 7 decimal digits of precision. This precision is important during training because gradients are tiny — small rounding errors accumulate and can derail learning.

After training, the model is frozen. You no longer need to compute gradients. The question becomes: how precisely do you need to represent each weight for the model to still work well during inference?

The answer, discovered empirically and somewhat surprisingly, is: not very precisely. Quantisation reduces each weight from 32 bits down to 16, 8, or even 4 bits. A 4-bit quantised 7B model requires only about 4 GB of memory — fitting comfortably on most gaming GPUs and many consumer laptops. The quality loss is real but often surprisingly small for practical tasks.

Precision levels explained

The quantisation landscape has its own terminology. Here are the levels you'll encounter:

FP32 (32-bit float) — full precision. Used for training. 4 bytes per parameter. A 7B model = 28 GB. Nobody runs inference at FP32 on consumer hardware.

FP16 / BF16 (16-bit float) — half precision. Standard for inference on server GPUs. 2 bytes per parameter. A 7B model = 14 GB. BF16 (Brain Float 16, developed by Google) has slightly different numerical properties and is preferred for training. FP16 is more common for inference.

INT8 / Q8 (8-bit integer) — 1 byte per parameter. A 7B model = 7 GB. Small quality loss for most tasks. Fits on a high-end consumer GPU.

Q4 (4-bit integer) — 0.5 bytes per parameter. A 7B model = ~4 GB. Moderate quality loss, model-dependent. Fits on a mid-range GPU or Apple Silicon Mac. The most common format for local use.

Q2 / Q3 (2–3 bit) — very aggressive compression. Models become noticeably degraded. Useful only for the most constrained hardware.

Why 4-bit works better than you'd expect

A 4-bit integer can represent only 16 distinct values (0 through 15). Squashing each weight — which during training was a precise float spanning a wide range — into one of 16 buckets sounds catastrophically lossy. And yet Q4 models perform at roughly 95–98% of their full-precision counterparts on most benchmarks. Why? Because neural networks have massive redundancy. No single weight is critical; patterns are distributed across billions of values. Small rounding errors in individual weights average out across the billions of operations in a forward pass.

The quality trade-off

Quantisation is not free. The quality loss is small on average but uneven across tasks. Tasks that require precise arithmetic — complex mathematics, exact date calculations, careful logical reasoning — degrade more noticeably under aggressive quantisation than tasks requiring general language fluency, summarisation, or creative writing.

The practical rule of thumb: Q4 or Q5 quantisation of a good model gives you most of the quality at a fraction of the memory cost. Q8 gives you nearly the full quality. Q2 and Q3 should only be used if you have no other option — the degradation becomes obvious in extended conversations.

It's also worth noting that a larger quantised model often beats a smaller full-precision model. A Q4 Llama 3 70B (requiring ~40 GB) substantially outperforms a full-precision Llama 3 8B (requiring ~16 GB) even though both might "fit" on a high-memory system. When choosing what to run locally, maximise the model size first, then choose the quantisation level that fits your memory.

GGUF: the format that made local AI possible

In 2023, a developer named Georgi Gerganov released a project called llama.cpp — a C++ implementation of the LLaMA model inference engine. To distribute quantised models in a format that llama.cpp could use, he created a file format called GGUF (GPT-Generated Unified Format, though the G later came to stand for Georgi).

GGUF became the de facto standard for distributing local LLMs. A single .gguf file contains everything needed to run a model: the architecture definition, the tokeniser vocabulary, and all the quantised weights. Download one file, point a compatible runtime at it, and the model runs. No Python environments, no dependency management, no GPU drivers to configure beyond basics.

On Hugging Face — the main repository for open AI models — GGUF variants of every major open model are typically available within hours of a new model release, uploaded by the community. The most prolific creator of these quantised versions is a user named "TheBloke" (now "bartowski"), who has uploaded thousands of quantised models that have been downloaded hundreds of millions of times.

llama.cpp: the engine underneath

llama.cpp is remarkable for what it is: a single C++ file, written by one developer in a few days in early 2023, that can run any GGUF-format model on almost any hardware. It supports CUDA (NVIDIA GPUs), Metal (Apple Silicon), ROCm (AMD GPUs), and CPU-only inference. It runs on Windows, Mac, Linux. It has no Python dependency.

The CPU-only mode is slow — a 7B model might generate two or three words per second on a modern laptop CPU — but it works. This matters enormously for accessibility: someone with no GPU can still run a local LLM. They'll wait a bit longer for responses, but the capability is there.

llama.cpp is also the foundation on which most local AI tools are built. Ollama, LM Studio, Jan, and many others are essentially user-friendly interfaces that sit on top of llama.cpp's inference engine. Understanding what llama.cpp is helps you understand what these tools are doing under the hood.

Distillation: teaching small models from big ones

Quantisation reduces the size of an existing model. Distillation takes a different approach: train a smaller model from scratch, but use a larger model's outputs as training signal rather than raw human-generated data.

The insight behind distillation is that a large model's output distribution contains more information than a simple right/wrong label. When a 70B model says "the answer is Paris, with high confidence, and London is somewhat plausible but wrong," a small model learning from that distribution learns more than it would from just seeing "the answer is Paris." The large model's uncertainty is itself a form of knowledge.

Microsoft's Phi series demonstrates distillation's power dramatically. Phi-3-mini (3.8B parameters) was trained on synthetic data generated by larger models. It outperforms many 7B models and holds its own against some 13B models on reasoning benchmarks. The model is small enough to run comfortably on a modern laptop — and genuinely useful for real tasks.

Real example
DeepSeek-R1 and the distillation breakthrough
China · January 2025

DeepSeek, a Chinese AI lab, released DeepSeek-R1 in January 2025 — a reasoning model that matched GPT-4o and Claude 3.5 Sonnet on many benchmarks, at a fraction of the reported training cost. The release included distilled versions ranging from 1.5B to 70B parameters, trained on R1's outputs.

The 7B and 14B distilled versions were particularly notable: small enough to run comfortably on consumer hardware, yet retaining much of the step-by-step reasoning capability of the full model. DeepSeek-R1-Distill-Qwen-14B became one of the most downloaded local models in the weeks after release. It demonstrated that distillation could preserve not just factual knowledge but complex reasoning strategies — something that hadn't been clearly established before.

Model families and size tiers

The open-weight model landscape has settled into recognisable size tiers, each with a different hardware profile:

1B–3B — fit easily on phones and very constrained hardware. Limited reasoning ability. Useful for specific narrow tasks: autocomplete, classification, short Q&A. Examples: Phi-3-mini, Llama 3.2 1B/3B, Gemma 2 2B.

7B–8B — the sweet spot for most consumer hardware. Require 4–8 GB VRAM at Q4. Capable of extended conversation, basic coding, summarisation, and general Q&A. Good enough for most everyday tasks. Examples: Llama 3.1 8B, Mistral 7B, Gemma 2 9B, Phi-3.5-mini.

13B–14B — noticeably better reasoning than 7B, require 8–12 GB VRAM at Q4. The first tier where mathematical reasoning becomes reliably useful. Examples: Phi-3-medium 14B, DeepSeek-R1-Distill-Qwen-14B.

32B–34B — require 20–25 GB VRAM at Q4. Approaching the quality of GPT-3.5 class cloud models. Examples: Qwen2.5 32B, DeepSeek-R1-Distill-Qwen-32B. Run well on a 32 GB Apple Silicon Mac.

70B — require 40–48 GB VRAM at Q4. Approach GPT-4-class quality for many tasks. Require two consumer GPUs or a Mac with 64+ GB RAM. Examples: Llama 3.3 70B, Qwen2.5 72B. This is where local AI becomes genuinely impressive.

How to choose a model for your hardware

The decision tree is simpler than it appears. Start with your available memory, then choose the largest model that fits at a reasonable quantisation level.

8 GB VRAM or RAM → 7B/8B model at Q4 (e.g. Llama 3.1 8B Q4, Mistral 7B Q4). Capable for general conversation and light coding.

16 GB unified memory (Apple M-series) → 7B at Q8 or 13B at Q4. Noticeable quality improvement over 8 GB options.

24 GB VRAM (RTX 4090 / 3090) → 13B at Q8 or 34B at Q4. The 34B tier is a substantial jump in quality.

32 GB unified memory (Apple M-series) → 32B/34B at Q4 or Q5. This is where local AI becomes impressive for serious use.

64 GB unified memory (Mac Studio / Mac Pro) → 70B at Q4. Approaches GPT-4 quality for many tasks. Chapter 07 covers this setup in detail.

2× consumer GPUs (48 GB total) → 70B at Q4. Requires model sharding across two GPUs — Ollama handles this automatically.

Model sharding
Splitting a model across multiple GPUs because it's too large to fit on one. Each GPU holds a portion of the model's layers. Requires fast inter-GPU communication. Ollama handles sharding automatically across multiple NVIDIA GPUs on the same machine. Slightly slower than a single-GPU setup due to communication overhead.

Context window trade-offs at small scale

On consumer hardware, the context window creates a direct memory trade-off. Every token in the context window requires memory to store its KV cache — the intermediate attention computations that allow efficient sequential generation. A 32,000-token context window uses substantially more memory than a 4,000-token one.

Ollama and llama.cpp both allow you to configure the context window size. The default is often conservative to save memory. If you're working with long documents and have spare memory, increasing the context window gives you a larger "working memory." If you're already at the edge of your memory, reducing it lets you run a larger or higher-precision model.

The practical implication: on a 16 GB Mac, you might run a 7B model with a 16,000-token context, or a 13B model with an 8,000-token context, depending on your use case. Long documents need context; complex reasoning needs model size. Know your task before choosing your configuration.

Where this is going

Quantisation and distillation are active research areas, not settled techniques. The frontier is moving fast in both directions.

On the compression side: 4-bit is no longer the floor. Research groups are demonstrating 2-bit and even 1-bit quantisation schemes that preserve more quality than integer rounding would suggest, using more sophisticated compression algorithms. The theoretical minimum for running a useful LLM locally is still being discovered.

On the model side: each new generation of models is more capable at smaller sizes. Llama 3.1 8B outperforms the original Llama 1 65B on most benchmarks. In two or three years, a model running on a phone will likely exceed the quality of today's best cloud models in many domains.

The direction is clear: capable AI is moving to the edge. Not because cloud AI is going away — frontier models will stay in data centres — but because a growing range of tasks are becoming achievable locally, privately, without a subscription, without latency, without sending your data to a third-party server. Chapter 07 walks you through setting this up.
