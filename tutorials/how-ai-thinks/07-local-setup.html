Chapter 07 · Practical setup

Running Your Own LLM

This chapter is practical. By the end of it you'll have a local LLM running on your machine, understand which model to pick for your hardware, and know what you can realistically do with it — and what you can't.

~45 min read 11 sections Practical setup

Why run locally at all?

Before investing time in setup, it's worth being clear about why local inference is worth doing. The cloud alternatives — ChatGPT, Claude, Gemini — are genuinely excellent and require no setup. If your goal is just to use AI effectively, a cloud subscription is the faster path.

Local LLMs make sense when one or more of these apply to you:

Privacy. Your data stays on your machine. Nothing is sent to a third-party server. For legal documents, medical notes, proprietary business information, personal journals — local inference means zero data leakage by design. No privacy policy to trust. No server logs of your conversations.

Cost at volume. Cloud APIs charge per token. At low usage, a subscription is fine. At high usage — processing thousands of documents, running automated pipelines, building tools for a team — local inference costs electricity, not per-query fees.

Offline access. Once downloaded, a local model works without internet. Useful on planes, in areas with poor connectivity, or when you need guaranteed availability.

Customisation. Local models can be fine-tuned on your own data, configured with custom system prompts, and integrated into local workflows in ways cloud APIs don't always support.

Curiosity and understanding. Running a model locally gives you a visceral sense of what these systems actually are. You see the model file, you watch it load into memory, you feel the inference speed vary with prompt length. It's the difference between using a car and opening the bonnet.

What you actually need

The minimum viable local LLM setup is lower than most people expect. Here's the honest picture for a 32 GB system:

HARDWARE
32 GB system — what's possible

On a Mac with 32 GB unified memory (M1/M2/M3/M4 Pro or Max): you can run a 32B model at Q4 comfortably, or a 13B at Q8. Apple Silicon's unified memory architecture means the GPU can use all 32 GB, not just a dedicated VRAM pool. This is genuinely the best value platform for local AI right now.

On a Windows/Linux machine with 32 GB RAM and an NVIDIA GPU with 12–16 GB VRAM: the GPU VRAM is the bottleneck, not total RAM. With a 16 GB GPU (e.g. RTX 4080), you can run a 13B model at Q4 fully on GPU. Larger models can partially offload layers to CPU RAM, but with significant speed penalty.

On a CPU-only machine with 32 GB RAM: inference will be slow (2–5 tokens per second for a 7B model), but it works. Useful for low-throughput tasks where you're not waiting in real time.

Storage: Model files range from 4 GB (7B Q4) to 40 GB (70B Q4). Keep 50–100 GB free on a fast SSD. Model load time from an NVMe SSD is a few seconds; from a spinning hard drive, painfully slow.

Software: Ollama (covered next) handles everything else. No manual driver configuration, no Python environment management, no CUDA setup beyond having up-to-date NVIDIA drivers.

Ollama: the easiest starting point

Ollama is the tool that made local LLMs accessible to non-engineers. It wraps llama.cpp in a clean interface, manages model downloads and storage, handles GPU detection and configuration, and exposes a local API that is compatible with OpenAI's API format — meaning any tool that works with ChatGPT's API can be pointed at Ollama with a URL change.

What Ollama does behind the scenes when you run a model: it downloads the GGUF file from its model library (hosted on its servers), detects your GPU capabilities, decides how many layers to offload to GPU vs CPU, loads the model into memory, and starts a local server at http://localhost:11434 that accepts requests. You don't need to know any of this — it just works.

Other good options exist: LM Studio has an excellent GUI, is good for browsing and downloading models from Hugging Face, and gives fine-grained control over inference settings. Jan is an open-source ChatGPT-style interface that also manages local models. GPT4All is particularly beginner-friendly. This guide focuses on Ollama because it's the most flexible for power users and developers, but any of these will work for getting started.

Installing and running your first model

The full setup takes about five minutes plus however long the model download takes.

STEP 1
Install Ollama

Go to ollama.com and download the installer for your platform (Mac, Windows, Linux). On Mac, drag to Applications. On Linux, one-line install:

# Linux install
curl -fsSL https://ollama.com/install.sh | sh
STEP 2
Pull and run a model

Open Terminal (Mac/Linux) or Command Prompt (Windows). Pull and run a model in one command:

# Pull and chat with Llama 3.2 3B (fast, ~2 GB)
ollama run llama3.2

# Or a larger model for a 32 GB system
ollama run qwen2.5:32b

# The first run downloads the model — subsequent runs are instant
STEP 3
Chat directly in terminal

After the model loads, you'll see a >>> prompt. Type your message and press Enter. That's it — you're talking to a local LLM.

>>> Explain gradient descent in plain English

Gradient descent is an optimisation algorithm used to find the minimum
of a function. Imagine you're standing on a hilly landscape in thick fog...
STEP 4 (optional)
Use the API

Ollama runs a local server you can query from any language or tool:

# Query via curl
curl http://localhost:11434/api/generate -d '{
"model": "qwen2.5:32b",
"prompt": "What is a transformer in AI?",
"stream": false
}'

Which model for a 32 GB system

For a 32 GB Mac (M-series), the best general-purpose choice in mid-2025 is Qwen2.5 32B at Q4 quantisation. It fits comfortably (uses about 20 GB), runs at 15–25 tokens per second on M3/M4 Pro chips, and its quality is genuinely impressive — capable of multi-step reasoning, complex coding, document analysis, and extended conversation.

If reasoning tasks (mathematics, logic, step-by-step problem solving) are your priority, DeepSeek-R1-Distill-Qwen-32B is the better choice. It uses the same memory footprint but is specifically optimised for reasoning tasks, often dramatically outperforming general-purpose models on complex problems.

For a 32 GB Windows/Linux machine where GPU VRAM is the constraint (not total RAM), the picture changes: a 16 GB GPU runs a 13B model smoothly on GPU, with larger models requiring CPU offloading (slower). The sweet spot is a 13B model fully in GPU VRAM rather than a 32B model split between GPU and CPU.

Open WebUI: a browser interface

Typing in a terminal is functional but not pleasant for extended use. Open WebUI gives you a ChatGPT-style browser interface running entirely locally. It connects to Ollama, shows all your installed models, persists conversation history, and supports file uploads for document analysis.

# Install Open WebUI via Docker (simplest method)
docker run -d -p 3000:8080 \
--add-host=host.docker.internal:host-gateway \
-v open-webui:/app/backend/data \
--name open-webui \
ghcr.io/open-webui/open-webui:main

# Then open http://localhost:3000 in your browser

If Docker is too much overhead, LM Studio is a native GUI alternative that doesn't require it — downloadable from lmstudio.ai and very polished.

Real use cases, honest assessments

Here's an honest assessment of where local LLMs (on a 32 GB system with a 32B model) are good, where they're acceptable, and where cloud models still win.

Local LLMs are excellent for: Drafting and editing text (emails, documents, reports). Summarising documents you paste in. Answering questions about content you provide. General Q&A on topics with training coverage. Basic to intermediate coding tasks (writing functions, explaining code, debugging). Brainstorming. Translation for major languages.

Local LLMs are acceptable for: Complex coding (they'll make mistakes on hard problems; cloud models make fewer). Extended reasoning chains (better with reasoning-optimised models like DeepSeek-R1). Multilingual tasks in non-English languages (quality drops for less-represented languages). Tasks requiring current information (knowledge cutoff applies; no internet access by default).

Cloud models still win for: Tasks requiring very long context (hundreds of pages of documents). Latest current information. Vision tasks (analysing images, charts) — though multimodal local models are improving. Maximum quality on hard tasks. Tasks where speed matters more than privacy.

The honest comparison to cloud

A local Qwen2.5 32B is roughly comparable to GPT-3.5 Turbo or Claude Haiku in quality — capable for a wide range of everyday tasks, noticeably below GPT-4o or Claude Sonnet for hard problems. That gap will close over time. For now: local is good enough for the majority of daily AI use cases, and private by design. For your hardest, most important tasks, cloud models still have an edge.

RAG: making the model know your documents

RAG (Retrieval-Augmented Generation) is how you make a local model knowledgeable about your own documents — your notes, your PDFs, your codebase — without retraining it.

The process: documents are split into chunks and converted to embeddings (numerical representations of meaning) using a small embedding model, then stored in a local vector database. When you ask a question, your question is also converted to an embedding, and the database retrieves the most semantically similar document chunks. Those chunks are included in the model's context window alongside your question.

Open WebUI has built-in RAG support — you can upload documents directly and the interface handles the embedding and retrieval. For more control, tools like Ollama + Chroma (a local vector database) + LangChain give you a full pipeline you can customise.

RAG is powerful but has limits: the retrieval step can miss relevant context that's phrased differently from your query, and very long documents require careful chunking strategy. It works well for factual retrieval ("what does clause 5.3 say?") and less well for synthesis across many documents ("summarise the key themes across all these reports").

Performance tips

A few practical adjustments that make a meaningful difference:

Set GPU layers explicitly. Ollama usually auto-detects, but for borderline cases you can force full GPU offload: OLLAMA_NUM_GPU=99 ollama run qwen2.5:32b. The 99 means "offload as many layers as fit in VRAM."

Keep the model loaded. The first prompt after loading takes longer because the model is being loaded into memory. Subsequent prompts are faster. Ollama keeps models loaded for 5 minutes by default; set OLLAMA_KEEP_ALIVE=1h if you're doing sustained work.

Use a shorter context window for speed. Longer context = slower generation. If you don't need 32K context, set a smaller value in the Modelfile or via the API: "num_ctx": 8192.

Close other applications. On Apple Silicon, other apps compete for the same unified memory pool. Closing Chrome and other memory-hungry apps before running a large model measurably improves speed.

Use an NVMe SSD. Model loading time scales with storage read speed. An NVMe SSD loads a 20 GB model in 5–10 seconds. A USB drive takes minutes.

Honest limitations

Running a local LLM is rewarding but comes with genuine limitations worth being clear about before you invest time.

Knowledge cutoff. The model knows nothing that happened after its training data cutoff. For Llama 3.3, that's roughly early 2024. For Qwen2.5, late 2024. For current news, prices, recent research — the model is wrong by default.

No internet. By default, local models can't search the web. You can add tools and agents that give them web access (some Open WebUI configurations support this), but it's not automatic.

Hallucinations. Local models hallucinate — generate plausible-sounding but wrong information — at rates comparable to cloud models of similar capability. Smaller models hallucinate more. Don't use local LLMs for tasks where factual accuracy is critical without verification.

Speed. Even on a fast Mac, a 32B model generates 15–25 tokens per second — about as fast as you can comfortably read. Cloud models are often 2–5x faster. For real-time interactive use, speed is fine. For batch processing thousands of documents, the difference matters.

None of these limitations are dealbreakers. They're just the honest picture of where the technology is right now. Local LLMs are genuinely useful, private, free to run, and improving rapidly. They're not magic, and they're not trying to be.

"Running a local model is like having a very knowledgeable colleague who works for free, never sleeps, never shares your secrets — and occasionally makes things up with alarming confidence." — practical summary from the local AI community
